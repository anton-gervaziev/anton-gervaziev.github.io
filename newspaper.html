The Situation

A 30-year-old newspaper distribution company. The sole operator managing the entire operation: subscriber routes, daily deliveries, billing across 6 physical distribution points.

Every morning the operator opened a DOS application from 1998. The Windows 98 machine it ran on was the only one that could run it. No documentation, no source code, no backup, no recovery plan. One hardware failure and the business stops completely.

On top of the hardware risk — no reporting, no remote access, no way to connect it to anything. The hardware was already failing. The system needed a full rebuild before the hardware failed completely.

What I Built

No codebase, no documentation, no schema. I sat with the operator and watched how the DOS system was used every day — what was clicked, what was remembered from habit, what the software didn't actually handle. Then I turned all of that into a data model and started building.

Route management
Subscriber assignments, daily route generation, change tracking across 6 locations.
Subscriber tracking
Sign-up, edition preferences, pauses, cancellations, reactivation. The whole cycle.
Billing
Monthly billing cycles, edition pricing, partial-month adjustments.
Multi-location
6 distribution points, each with their own routes and stock. One place to see all of it.
Data migration
Got subscriber and route data out of the DOS exports and into the new system by hand.
UI design
Designed around how the operator actually works — not how software usually works.
Print server
Raspberry Pi running a print server via Tailscale — accessible from phone and laptop, anywhere. Running for 6+ months.

Development Process

Phase 1

Discovery & Reverse Engineering

Spent days watching the owner use the DOS system. A lot of the business rules weren't in the software — they were in the operator's head.

Phase 2

Data Model Design

Built the database schema from nothing. The DOS data structure made no sense on paper — had to figure out what the tables and relationships actually were from how the app behaved.

Phase 3

Core Module Development

Route management first since everything depends on it, then subscribers, billing, multi-location. Checked each piece with the owner before moving on.

Phase 4

Data Migration & Cutover

Migrated all operational data from the DOS system. Ran parallel operations to validate, then retired the DOS machine. Zero data loss.

Ongoing

Support & Maintenance

System is in daily use. Bug fixes, feature requests, adjustments as the business evolves.

Technical Challenges

01

No Documentation Anywhere

No docs, no comments, original developer unreachable. Every rule discovered through observation — including rules the owner followed unconsciously.

02

Opaque Data Format

DOS stored data in proprietary binary format. Reverse-engineered file structure, wrote custom parsers, validated every migrated record.

03

Sole Operator

One person uses the system. UI had to match their mental model precisely, not generic UX patterns. If it felt wrong to them, it was wrong.

04

Business Continuity Risk

Single point of failure for a 30-year business. Migration had to be bulletproof — no fallback if data was lost or corrupted.
